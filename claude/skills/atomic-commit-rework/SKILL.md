# atomic-commit-rework

あなたはGit履歴を再編する担当です。

現在のブランチでは、要求された機能の実装と動作確認が完了しています。
これから、現在の変更内容を論理的かつatomicな単位のコミットへ再編してください。

## 目的

レビュー、revert、bisect、cherry-pickが容易になるように、変更を意味のある最小単位へ分割してください。

ここでいうatomic commitとは、単にファイル数やdiffサイズが小さいコミットではなく、以下を満たすコミットです。

* 1つの明確な目的だけを持つ
* そのコミット単体で変更意図を説明できる
* 可能な限り、そのコミット時点でビルドとテストが成功する
* 後続コミットの変更を前提にしなくても、リポジトリが論理的に整合している
* 独立してrevertまたはcherry-pickできる
* リファクタリングと振る舞い変更を可能な限り分離する

## 最重要制約

* 作業開始前に、現在のHEADを指すバックアップブランチまたはタグを作成すること
* リモートブランチへpushしないこと
* force pushしないこと
* main、master、developなど共有ブランチを変更しないこと
* 最終的なワーキングツリーの内容を、再編前の状態から変更しないこと
* 機能追加、バグ修正、追加リファクタリングを行わないこと
* コミット再編に不要なコード変更を行わないこと
* 変更内容を失う可能性がある操作を、バックアップ作成前に実行しないこと

## 作業手順

### 1. 現状確認

最初に以下を確認してください。

* 現在のブランチ名
* ベースブランチまたは分岐元コミット
* ベースとの差分
* 現在のコミット履歴
* staged、unstaged、untrackedファイルの有無
* 実行可能なビルド、lint、テストコマンド
* 変更によって実現された要求事項

ベースコミットが明確でない場合は、merge-base、ブランチ履歴、リモート追跡ブランチを調査して合理的に特定してください。

### 2. バックアップ

現在のHEADを指すバックアップブランチを作成してください。

バックアップ名の例:

`backup/atomic-rework-<timestamp>`

未コミットの変更がある場合は、それらも失われない方法で保護してください。

### 3. 変更の分析

ベースとの差分を、ファイル単位ではなく変更意図単位で分析してください。

以下の観点で変更を分類してください。

* 基盤、型、インターフェース、スキーマの追加
* ドメインロジック
* アプリケーションロジック
* 外部サービスやインフラストラクチャとの統合
* 設定変更
* テスト追加・変更
* ドキュメント
* 純粋なリファクタリング
* 自動生成物
* フォーマットのみの変更
* 不要になったコードの削除

同じファイル内に複数の論理的変更が含まれる場合は、hunk単位で分割してください。

### 4. コミット計画

実際に履歴を書き換える前に、コミット計画を提示してください。

各コミットについて、以下を記載してください。

1. コミットタイトル
2. 変更目的
3. 含めるファイルまたはhunk
4. 含めない関連変更
5. 依存する先行コミット
6. 実行する検証
7. 単独でrevertまたはcherry-pick可能か

コミット順序は、後続変更が自然に積み上がる依存順にしてください。

原則として、以下の順序を検討してください。ただし、実際の依存関係を優先してください。

1. 前提となる型、インターフェース、設定
2. 内部実装
3. 統合処理
4. テスト
5. ドキュメントと補助的変更

テストは原則として対象実装と同じコミットに含めてください。
ただし、既存挙動を固定するcharacterization testや、意図的に失敗するテストを先に導入する明確な理由がある場合は分離できます。

### 5. コミット再編

コミット計画に従って、現在の変更を再構築してください。

#### 既定の再編手法

以下の手順を既定としてください。

1. バックアップブランチが存在することを確認する
2. `git reset <ベースコミット>` を実行する。HEADはベースコミットへ戻り、全変更はunstagedとしてworking treeに残る
3. コミット計画の順序どおりに、論理単位のstageとcommitを繰り返す

`git rebase -i` は使用しないでください。editorの対話操作を前提とするため、非対話環境では失敗します。やむを得ず使用する場合のみ、`GIT_SEQUENCE_EDITOR` に非対話コマンドを設定してください。

ベースコミット以降に新規追加されたファイルは、reset後にuntrackedになります。ファイル全体を1コミットへ含める場合は `git add <ファイル>` で追加してください。新規ファイルをhunk分割する必要がある場合は、`git add -N <ファイル>` で追跡対象にしてから、次のhunk単位のstage手順を使用してください。

#### hunk単位のstage手順

`git add -p` は対話コマンドのため使用しないでください。同一ファイル内に複数の論理変更が混在する場合は、以下の手順でhunk単位にstageしてください。

1. `git diff -- <ファイル>` の出力をpatchファイルへ保存する
2. patchファイルを編集し、stageするhunkだけを残す
3. `git apply --cached <patchファイル>` でindexへ適用する
4. `git diff --cached` でstage内容がコミット計画と一致することを確認してからcommitする

patchの編集は、hunk全体を残すか削除するかの単位で行ってください。hunk内部の行を編集した場合は、行数の不整合を避けるため `git apply --cached --recount` を使用してください。

以下を守ってください。

* 無関係な変更を同じコミットに混在させない
* 同一の論理変更を複数コミットへ不必要に分散しない
* renameと内容変更は、レビュー容易性が向上する場合は分離する
* 大規模なフォーマット変更と機能変更は分離する
* コード生成物は、その生成元変更と同じコミットに含めるか、明確な理由を持つ独立コミットにする
* 一時的な修正、試行錯誤、fixup、WIPコミットを最終履歴に残さない
* コミットメッセージから変更理由が理解できるようにする

コミットメッセージは、このリポジトリの既存規約に従ってください。
明確な規約が存在しない場合は、Conventional Commits形式を使用してください。

例:

`feat(auth): validate token audience`

本文が必要な場合は、実装内容の列挙ではなく、変更理由、制約、重要な判断を記載してください。

### 6. コミット単位の検証

可能な限り、各コミットについて以下を実行してください。

* formatter
* lint
* type check
* unit test
* integration test
* build

#### 検証方法

reset方式では、コミット作成直後のworking treeに後続コミット予定の変更が残っています。そのまま検証しても、そのコミット単体の検証にはなりません。以下のいずれかの方法で検証してください。

方法A: コミットごとに退避して検証する

1. コミット作成直後に `git stash push -u` で残りの変更を退避する
2. 検証コマンドを実行する
3. `git stash pop` で変更を復元し、次のコミット作業へ進む

検証がファイルを生成または変更する場合は、`git stash pop` の前にworking treeを検証前の状態へ戻してください。

方法B: 全コミット作成後に一括で検証する

すべてのコミットを作成した後、`git rebase <ベースコミット> --exec '<検証コマンド>'` を実行してください。各コミットをcheckoutした状態で検証コマンドが順に実行され、失敗したコミットで停止します。停止した場合はコミット境界を見直し、修正後に `git rebase --continue` で再開してください。

すべての検証を毎コミット実行するコストが高い場合は、方法Bを使用するか、変更範囲に対応する最小限の検証を各コミットで実行し、最終状態で全体検証を実行してください。

各コミットでビルドやテストを成功させられない場合は、安易に無視せず、次のいずれかを行ってください。

* コミット境界を見直す
* 依存する変更を同じコミットへ統合する
* そのコミット単独では検証できない理由を明示する

テストを意図的に失敗させるコミットは、明示的に要求されていない限り作成しないでください。

### 7. 最終確認

再編完了後、以下を確認してください。

* 再編前バックアップと再編後HEADの最終ツリーが同一であること
* 意図しないファイル差分がないこと
* working treeがcleanであること
* 全体のビルド、lint、テストが成功すること
* コミット順序が依存関係に沿っていること
* 各コミットが1つの論理的目的を持つこと
* コミットメッセージが変更内容と一致していること

最終ツリーの同一性は、可能であれば以下のような方法で機械的に確認してください。

* tree object IDの比較
* `git diff <backup-branch> HEAD`
* 必要に応じて生成物や除外ファイルも含めた追加確認

`git diff <backup-branch> HEAD`は、履歴ではなく最終ファイル内容について差分がない状態にしてください。

## 完了報告

作業完了後、以下を報告してください。

1. 作成したバックアップブランチ名
2. 特定したベースコミット
3. 再編前と再編後のコミット数
4. 再編後のコミット一覧
5. 各コミットの目的
6. 各コミットで実行した検証
7. 最終状態で実行した検証
8. 再編前後の最終ツリーが同一であることの確認結果
9. 実行しなかった検証と、その理由
10. 残っているリスクまたは判断が必要な事項

## 判断基準

コミット数を増やすこと自体を目的にしないでください。

以下の場合は、無理に分割せず1コミットへまとめてください。

* 分割すると中間コミットがビルド不能になる
* 変更同士が強く結合しており、単独で意味を持たない
* 独立したrevertやcherry-pickが現実的でない
* 分割によって変更意図がかえって理解しにくくなる

反対に、以下は原則として分離してください。

* 振る舞いを変えないリファクタリングと機能変更
* フォーマット変更とロジック変更
* 複数の独立したバグ修正
* 独立してrevertできる機能追加
* 本質的に無関係な設定変更
* 変更目的が異なるドキュメント更新

不明点があっても、まずリポジトリと履歴から判断材料を収集してください。
破壊的操作が必要になる場合のみ、実行前に停止して確認してください。
